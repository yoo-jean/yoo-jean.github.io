# 1. Lock(락)이란?

Lock은 한 마디로 말하면

> "내가 지금 이 데이터를 쓰고 있으니까 잠깐만 기다려줘!"

라고 말하는 장치다.

---

## 왜 필요할까?

은행 계좌를 예로 들어보자.

현재 잔액: 100만원

- 트랜잭션 A → 100만원 출금
- 트랜잭션 B → 100만원 출금

동시에 실행되면 어떻게 될까?

1. A가 잔액 조회 → 100만원
2. B도 잔액 조회 → 100만원
3. A 출금 → 잔액 0원
4. B 출금 → 잔액 0원

→ 실제로는 200만원이 빠져야 하는데 100만원만 빠짐  
→ 데이터가 깨짐 (정합성 문제)

그래서 Lock이 필요하다.

---

# 2. 공유락 (Shared Lock, S-Lock)

## 쉽게 말하면

> "읽는 건 같이 봐도 돼. 하지만 수정은 안 돼."

---

##  상황 예시

게시판 글을 여러 사람이 동시에 읽는 상황을 생각해보자.

### 트랜잭션 A

``` xml
SELECT * FROM post WHERE id=1;
```

### 트랜잭션 B

``` xml
SELECT * FROM post WHERE id=1;
```


 둘 다 읽기 → 문제 없음  
 공유락은 여러 개 가능

---

##  그런데 누가 수정하려고 하면?

### 트랜잭션 C
``` xml
UPDATE post SET title='수정' WHERE id=1;
```


→ 지금 읽고 있는 사람이 있으니  
→ C는 기다려야 한다.

---

##  정리

| 구분 | 공유락 |
|------|--------|
| 여러 명 읽기 | 가능 |
| 수정 | 불가능 |
| 비유 | 도서관에서 같은 책 여러 명이 읽는 상황 |

---

# 3. 배타락 (Exclusive Lock, X-Lock)

##  쉽게 말하면

> "지금 내가 수정 중이니까 아무도 건들지 마."

---

##  상황 예시

트랜잭션 A:
``` xml
UPDATE users SET name='kim' WHERE id=1;
```

→ 배타락 획득

그 순간,

### 트랜잭션 B
``` xml
SELECT * FROM users WHERE id=1;
```


→ 읽기도 못 함  
→ 기다려야 함

---

##  왜 이렇게 강하게 막을까?

수정 중에 누군가 읽으면

- 아직 반영 안 된 값 읽을 수 있음
- Dirty Read 발생 가능

---

##  정리

| 구분 | 배타락 |
|------|--------|
| 읽기 | 불가능 |
| 수정 | 불가능 |
| 비유 | 화장실 안에 사람 들어가서 문 잠근 상태 |

---

# 4. 공유락 vs 배타락 한 번에 이해하기

| 상황 | 공유락 | 배타락 |
|------|--------|--------|
| 여러 명 읽기 | 가능 | 불가능 |
| 읽는 중 수정 | 불가능 | - |
| 수정 중 읽기 | - | 불가능 |
| 수정 중 수정 | 불가능 | 불가능 |

---

# 5. Deadlock (교착상태) 쉽게 이해하기

##  상황 비유

두 사람이 서로 열쇠를 하나씩 들고 있다.

- A는 1번 문 열쇠 가지고 있음
- B는 2번 문 열쇠 가지고 있음

그런데:

- A는 2번 문 열고 싶음 (B가 가진 열쇠 필요)
- B는 1번 문 열고 싶음 (A가 가진 열쇠 필요)

→ 서로 기다림  
→ 영원히 안 끝남  
→ 이게 Deadlock

---

##  DB 상황 예시

트랜잭션 A:
> 1번 행 UPDATE → 성공  
> 2번 행 UPDATE → 대기

트랜잭션 B:
> 2번 행 UPDATE → 성공  
> 1번 행 UPDATE → 대기


→ 서로 기다림  
→ 교착상태 발생

---

##  해결 방법

1. 항상 같은 순서로 UPDATE
2. 트랜잭션 짧게 유지
3. 필요한 데이터만 잠그기
4. 인덱스 잘 설계하기

---

# 최종 정리

> Lock은 동시성 환경에서 데이터 정합성을 보장하기 위한 장치입니다.  
> 공유락은 읽기 시 여러 트랜잭션이 동시에 접근 가능하지만 수정은 제한되고,  
> 배타락은 수정 시 다른 트랜잭션의 읽기와 쓰기를 모두 막습니다.  
> 교착상태는 서로의 락을 기다리는 상황이며, 트랜잭션 순서를 일관되게 유지하여 예방할 수 있습니다.
