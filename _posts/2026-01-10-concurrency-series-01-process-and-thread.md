동시성(Concurrency)을 이해하기 위해 가장 먼저 짚고 넘어가야 할 개념은  
**프로세스(Process)** 와 **스레드(Thread)** 이다.

이 둘은 모두 실행 단위이지만,  
 *무엇을 기준으로 자원을 할당하고, 어떻게 실행 흐름을 관리하느냐* 에서 본질적인 차이가 있다.

---

## 1.1 프로세스와 스레드

### 프로세스란 무엇인가

프로세스는 **실행 중인 프로그램의 인스턴스**이다.  
운영체제 관점에서 프로세스는 **자원 할당의 최소 단위**이다.

운영체제는 각 프로세스마다 다음 자원을 **독립적으로** 할당한다.

- Code 영역
- Data 영역
- Heap 영역
- Stack 영역
- 파일 디스크립터
- 기타 OS 자원

#### 특징
- 프로세스 간 메모리 공유 ❌
- 하나의 프로세스가 죽어도 다른 프로세스에 영향이 적음
- 안정성은 높지만 생성/전환 비용이 큼

---

### 스레드란 무엇인가

스레드는 **프로세스 내부에서 실행되는 최소 실행 단위**이다.  
하나의 프로세스는 **여러 개의 스레드**를 가질 수 있다.

스레드는 다음과 같은 특징을 가진다.

- Code / Data / Heap 영역을 **공유**
- Stack 영역은 **스레드마다 독립**
- 같은 프로세스 내 스레드끼리는 빠른 통신 가능

#### 특징
- 자원 공유로 인한 동시성 문제 발생 가능
- Context Switching 비용이 프로세스보다 작음
- 높은 성능과 응답성 제공

---

### 프로세스 vs 스레드 차이

| 구분 | 프로세스 | 스레드 |
|---|---|---|
| 개념 | 실행 중인 프로그램 | 실행 흐름 |
| 자원 할당 | 독립적 | 공유 |
| 메모리 | 완전 분리 | Heap/Data 공유 |
| 안정성 | 높음 | 상대적으로 낮음 |
| 생성 비용 | 큼 | 작음 |
| 통신 방식 | IPC 필요 | 메모리 공유 |

---

### 메모리 구조 차이

- **프로세스**
  - 각 프로세스마다 독립적인 가상 메모리 공간
  - 다른 프로세스 접근 불가

- **스레드**
  - 같은 프로세스 내 스레드는 Heap/Data 공유
  - Stack만 개별 소유

이 구조 때문에 **동시성 문제(Race Condition)** 가 발생한다.

---

### Context Switching

Context Switching은 **현재 실행 중인 작업의 상태를 저장하고,  
다른 작업으로 실행 흐름을 전환하는 과정**이다.

- 프로세스 전환
  - 메모리 매핑 변경
  - 비용 큼

- 스레드 전환
  - 같은 메모리 공간 사용
  - 상대적으로 비용 작음

따라서 성능 관점에서는   
**멀티프로세스보다 멀티스레드가 유리**한 경우가 많다.

---

## 1.2 스레드는 왜 만들어졌는가

### 단일 실행 흐름의 한계

초기의 프로그램은 **하나의 실행 흐름**만으로 동작했다.

이 구조에는 다음과 같은 한계가 있다.

- I/O 작업 중 CPU 유휴 상태 발생
- 사용자 입력에 대한 응답 지연
- 전체 프로그램이 하나의 작업에 종속됨

---

### 병렬성(Parallelism) vs 동시성(Concurrency)

- **병렬성 (Parallelism)**
  - 여러 작업이 실제로 동시에 실행됨
  - 멀티코어 환경에서 가능

- **동시성 (Concurrency)**
  - 여러 작업이 동시에 실행되는 것처럼 보이게 처리
  - 실행 흐름을 빠르게 전환

Java에서 다루는 핵심 문제는 대부분 **동시성**이다.

---

### 응답성과 자원 효율성

스레드는 다음 목적을 위해 등장했다.

- 사용자 응답성 향상
- CPU 자원 활용 극대화
- I/O 대기 시간 동안 다른 작업 수행

즉, 스레드는   
**성능을 높이기 위한 수단이 아니라, 자원을 효율적으로 쓰기 위한 설계 도구**이다.

---

## 정리

- 프로세스는 자원 관리의 단위
- 스레드는 실행 흐름의 단위
- 스레드는 성능과 응답성 문제를 해결하기 위해 등장
- 동시성 문제는 공유 자원에서 발생
