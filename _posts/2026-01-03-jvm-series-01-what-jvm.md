## 들어가며
자바 개발을 하다 보면 자연스럽게 `JVM`, `GC`, `Heap` 같은 단어를 접하게 된다.  
하지만 막상 **"JVM이 무엇인가요?"** 라는 질문을 받으면  
어디서부터 설명해야 할지 막막해지는 경우가 많다.

이 글에서는 **JVM이 왜 필요한지**,  
**무엇을 담당하는지**, **자바 프로그램이 어떻게 실행되는지**를  
큰 그림 중심으로 정리해본다.

---

## JVM이란 무엇인가
**JVM(Java Virtual Machine)** 은 자바 프로그램을 실행하기 위한 **가상 머신**이다.

조금 더 정확히 말하면,
- 자바 바이트코드(`.class`)를 실행하고
- 메모리를 관리하며
- 운영체제와 애플리케이션 사이에서 중간 역할을 수행하는
  **런타임 환경(Runtime Environment)** 이다.

---

## 왜 JVM이 필요한가
자바의 가장 큰 특징 중 하나는 다음 문장으로 요약된다.

> Write Once, Run Anywhere

이는 자바 코드가  
운영체제(Windows, Linux, macOS)에 직접 의존하지 않고  
**JVM 위에서 실행되기 때문에 가능**하다.

### JVM이 없었다면
- OS마다 다른 바이너리를 만들어야 했고
- 메모리 관리, 스레드 관리도 개발자가 직접 처리해야 했을 것이다

👉 JVM은 **플랫폼 독립성 + 안정성**을 제공한다.



---

## JDK / JRE / JVM의 차이

자바를 이야기할 때 자주 함께 등장하는 용어가  
**JDK, JRE, JVM** 이다.  
이 셋은 포함 관계로 이해하면 가장 쉽다.

---

### JVM (Java Virtual Machine)
JVM은 자바 프로그램을 실행하는 **가상 머신**이다.

- `.class` 바이트코드를 실행
- 메모리 관리 (Heap, Stack 등)
- Garbage Collection 수행
- 스레드 관리

👉 **실행 담당자**  
👉 자바 프로그램이 실제로 돌아가는 환경

---

### JRE (Java Runtime Environment)
JRE는 **자바 실행에 필요한 환경 묶음**이다.

구성 요소:
- JVM
- Java 표준 라이브러리 (`java.lang`, `java.util` 등)

👉 **실행만 가능**
- 자바 프로그램 실행 ✅
- 자바 컴파일 ❌

---

### JDK (Java Development Kit)
JDK는 **자바 개발을 위한 전체 도구 세트**다.

구성 요소:
- JRE (→ JVM 포함)
- 컴파일러 (`javac`)
- 디버거 (`jdb`)
- 각종 개발 도구 (`jps`, `jstack`, `jmap` 등)

👉 **개발 + 실행 모두 가능**

---

### 포함 관계 정리

JDK > JRE  
JDK > JVM


- JVM: 실행 엔진
- JRE: 실행 환경
- JDK: 개발 도구 세트

---

### 언제 무엇을 사용할까

| 상황 | 필요한 것 |
|----|----|
| 자바 프로그램 실행 | JRE |
| 자바 개발 | JDK |
| 자바 코드 실행 핵심 | JVM |

실제 개발 환경에서는  
👉 **JDK 하나만 설치**하면 충분하다.

---

### 한 줄 요약 
> JVM은 실행, JRE는 실행 환경, JDK는 개발 도구까지 포함한 패키지다.

---

## JVM의 주요 역할

### 1. 바이트코드 실행
- `.java` 파일 → 컴파일 → `.class` 파일
- JVM은 이 **바이트코드**를 해석(Interpreter)하거나
- JIT 컴파일러를 통해 기계어로 변환해 실행한다

---

### 2. 메모리 관리
JVM은 실행 중인 자바 프로그램의 메모리를 직접 관리한다.
- 객체 생성
- 객체 이동
- 더 이상 사용되지 않는 객체 제거(GC)

👉 개발자가 `free()`를 호출하지 않아도 되는 이유다.

---

### 3. Garbage Collection
- 더 이상 참조되지 않는 객체를 자동으로 정리
- 메모리 누수 위험 감소
- 안정적인 장기 실행 서비스 가능

---

### 4. 스레드 관리
- Java Thread를 OS Thread와 매핑
- 멀티스레드 환경에서도 안정적인 실행 보장

---

## Java 프로그램 실행 흐름

1. 개발자가 `.java` 파일 작성
2. `javac` 컴파일러가 `.class` 파일 생성
3. JVM의 Class Loader가 클래스 로딩
4. Runtime Data Area에 메모리 할당
5. Execution Engine이 바이트코드 실행

이 과정 전체를 **JVM이 책임진다**.

---

## JVM은 어디까지 책임질까
JVM이 모든 것을 대신해주는 것은 아니다.

| 구분 | 책임 주체 |
|----|----|
| 메모리 할당/해제 | JVM |
| 객체 생명주기 | JVM |
| 비즈니스 로직 | 개발자 |
| 애플리케이션 구조 | 개발자 |
| 성능 설계 | 개발자 + JVM |


---

## JVM을 왜 알아야 할까

### 1. 성능 이슈 대응
- OutOfMemoryError
- GC 지연
- 응답 속도 저하

👉 대부분 JVM과 연결된다.

---

### 2. 면접 단골 주제
- JVM 메모리 구조
- GC 동작 방식
- Heap과 Stack 차이

👉 **이해 기반 설명이 중요**

---

### 3. Spring, 서버 개발의 기반
Spring, Spring Boot, WAS(Tomcat)  
모두 JVM 위에서 동작한다.

👉 JVM은 **모든 Java 서버 개발의 바닥**이다.

---

## 한 줄 요약
> JVM은 자바 바이트코드를 실행하고, 메모리와 스레드를 관리하는 자바 실행 환경이다.

---

## 마무리
JVM은 처음엔 추상적으로 느껴졌으나,  
추후에는 **자바 코드가 실제로 어떻게 움직이는지**가 보이기 시작하도록 해보자.

JVM을 깊이 앎을 통해  
“자바를 쓴다”에서  
“자바를 이해한다”로 한 단계 올라가고 싶다.
