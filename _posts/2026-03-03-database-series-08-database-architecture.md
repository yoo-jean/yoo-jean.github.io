# 1. 클러스터링(Clustering)이란?

클러스터링은 **여러 대의 서버를 하나의 시스템처럼 묶어 운영하는 구조**이다.

즉, DB 서버가 1대가 아니라 2대 이상 존재하며  
**한 서버에 장애가 발생해도 서비스가 멈추지 않도록 만드는 고가용성(HA) 구성 방식**이다.

---

## 왜 필요한가?

DB 서버가 1대뿐이라면:

- 서버 장애 = 전체 서비스 중단
- 로그인 / 결제 / 조회 전부 멈춤
- 기업 입장에서는 막대한 손실

따라서 여러 대로 구성하여  
**하나가 죽어도 다른 서버가 즉시 대체하도록 만드는 것**이 클러스터링이다.

---

## 핵심 목적

### 1) 고가용성 (High Availability, HA)
항상 시스템이 살아있도록 유지하는 것

### 2) 장애 대응 (Failover)
장애 발생 시 다른 서버가 자동으로 역할을 이어받음

- Failover → 대체 서버 전환
- Failback → 복구 후 원래 상태로 복귀

---

## 구성 요소

- Node : 클러스터에 속한 각 서버
- Health Check : 서버 상태 감시 (heartbeat)
- VIP(가상 IP) : 클라이언트가 바라보는 대표 주소

---

## 종류

### Active - Standby
- 한 대만 동작
- 한 대는 대기

✔ 안정성 높음  
⚠ 자원 활용도 낮음

---

### Active - Active
- 여러 대가 동시에 동작

- 자원 활용도 높음  
- 데이터 동기화 복잡

---

# 2. 리플리케이션(Replication)이란?

리플리케이션은 **데이터를 다른 서버에 복제하는 구조**이다.

구조:

Primary → Replica

Primary는 쓰기 담당  
Replica는 복제 데이터를 유지

---

## 왜 필요한가?

### 1) 읽기 부하 분산

대부분의 서비스는 읽기 트래픽이 훨씬 많다.

예:
- 게시글 조회 1000번
- 글 작성 10번

읽기를 Replica로 분산하면  
Primary의 부하를 줄일 수 있다.

---

### 2) 백업 및 장애 대응

Primary 장애 발생 시  
Replica를 승격(promote)하여 Primary로 사용 가능

---

## 동작 방식

### 1) 비동기 복제 (Asynchronous)

- Primary가 먼저 커밋
- 나중에 Replica에 전달

- 성능 좋음  
- 복제 지연(Replication Lag) 발생 가능

---

### 2) 동기 복제 (Synchronous)

- Replica까지 저장 완료되어야 커밋

- 데이터 일관성 보장  
- 성능 저하 가능

---

## Replication Lag이란?

Primary와 Replica 간 데이터 반영 시간 차이.

문제 예시:

1. Primary에 INSERT
2. 바로 Replica에서 조회
3. 데이터가 없음

→ 실시간성이 중요한 서비스에서는 주의 필요

---

# 3. 파티셔닝(Partitioning)이란?

파티셔닝은 **하나의 테이블을 논리적으로 여러 조각으로 나누는 것**이다.

중요한 점:
> DB는 여전히 하나이며, 테이블만 나뉜다.

---

## 왜 필요한가?

대용량 테이블 (예: 1억 건)

WHERE date = '2026-03-01'

→ 전체를 다 검색하면 느림

날짜 기준으로 나눠두면  
해당 파티션만 조회 가능

---

## 종류

### 1) Range Partition
범위 기준 분할 (날짜, ID)

예:
2025 데이터  
2026 데이터

---

### 2) List Partition
특정 값 기준 분할

예:
서울 / 부산 / 제주

---

### 3) Hash Partition
해시 함수 기반 균등 분할

---

## 장점

- 대용량 테이블 성능 개선
- 관리 용이 (오래된 파티션 삭제 등)

---

## 단점

- 설계 미흡 시 성능 이점 없음
- 잘못된 조건 사용 시 전체 스캔

---

# 4. 샤딩(Sharding)이란?

샤딩은 **데이터를 서버 단위로 물리적으로 분리하는 것**이다.

즉, DB 자체를 나눈다.

예:

User 1~1000 → DB1  
User 1001~2000 → DB2

---

## 왜 필요한가?

트래픽이 너무 커서  
한 DB 서버로 감당 불가한 경우

→ 서버를 여러 대로 나눠 수평 확장

---

## 특징

- 수평 확장 (Scale-Out)
- 대규모 서비스 필수 구조

---

## 단점

### 1) 조인 어려움
DB가 다르면 JOIN 불가

→ 애플리케이션 레벨 처리 필요

---

### 2) 분산 트랜잭션 복잡
2개 DB에 동시에 커밋?

→ 2PC 등 필요
→ 구현 난이도 높음

---

### 3) 재샤딩 문제
데이터 증가 시 재분배 매우 어려움

---

# 5. 한눈에 비교

| 개념 | 목적 | 데이터 위치 | 확장 방식 |
|------|------|--------------|------------|
| Clustering | HA | 동일 데이터 | 장애 대비 |
| Replication | 읽기 분산 | 복제 데이터 | 부하 분산 |
| Partitioning | 성능 | 동일 DB 내부 | 논리적 분할 |
| Sharding | 확장 | 서로 다른 DB | 수평 확장 |

---

#  핵심 질문

### Q1. 파티셔닝과 샤딩의 차이는?

A.  
파티셔닝은 하나의 DB 내부에서 테이블을 나누는 것이고,  
샤딩은 DB 자체를 여러 대로 나누는 것이다.

---

### Q2. 리플리케이션의 단점은?

A.
- 복제 지연
- 데이터 불일치 가능성
- 쓰기 분산 불가

---

### Q3. 샤딩의 가장 큰 어려움은?

A.
- 분산 트랜잭션 처리
- 재샤딩
- 운영 복잡성 증가

---

#  전체 한 줄 정리

- 클러스터링 → 장애 나도 안 멈추게
- 리플리케이션 → 읽기 분산
- 파티셔닝 → 대용량 테이블 성능 개선
- 샤딩 → 초대형 서비스 확장 전략
