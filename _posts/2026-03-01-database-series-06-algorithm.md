# 1. 인덱스는 왜 필요한가?

## 테이블 풀 스캔의 문제

테이블에서 조건에 맞는 데이터를 찾을 때 인덱스가 없으면 전체를 모두 읽는 Full Scan이 발생한다.

- 데이터가 1,000만 건인 경우
- 조건 1건을 찾기 위해 1,000만 건을 모두 읽을 수 있다

따라서 DB는 빠른 탐색을 위해 인덱스라는 별도의 구조를 사용한다.

## 인덱스의 정의

인덱스는 정렬된 키를 기반으로 빠르게 탐색하기 위한 자료구조이며, 대부분의 RDBMS는 B-Tree 계열 구조를 사용한다.

---

# 2. B-Tree (B Tree)

## 정의

B-Tree는 디스크 환경에서 탐색을 빠르게 하기 위해 설계된 균형 트리 구조이다. 한 노드에 여러 개의 키를 저장한다.

디스크 I/O 비용은 매우 크기 때문에 한 번 읽을 때 최대한 많은 키를 담아 트리 높이를 줄이는 것이 핵심이다.

## 특징

- 균형 트리 구조이다
- 한 노드에 여러 키를 저장한다
- fan-out이 커서 트리 높이가 낮다
- 탐색, 삽입, 삭제의 시간 복잡도는 O(log N)이다

---

# 3. B+Tree

## 정의

B+Tree는 B-Tree의 변형 구조이다. 데이터(레코드 위치)는 리프 노드에만 저장하며, 리프 노드들은 연결 리스트 형태로 연결된다.

## 범위 검색에 강한 이유

```sql
SELECT * FROM orders
WHERE created_at BETWEEN '2026-03-01' AND '2026-03-03';
```

1. 시작 지점을 탐색한다
2. 리프 노드를 따라 순차적으로 읽는다

이로 인해 범위 검색에 매우 효율적이다.

## B-Tree vs B+Tree 비교

| 구분 | B-Tree | B+Tree |
|---|---|---|
| 데이터 저장 위치 | 내부/리프 가능 | 리프 노드에만 저장 |
| 범위 검색 | 상대적으로 불리 | 매우 유리 |
| 리프 노드 구조 | 단순 정렬 | 정렬 + 연결 리스트 |
| 실무 사용 | 일부 | 대부분의 RDBMS |

# 4. 클러스터드 인덱스 vs 논클러스터드 인덱스

## 클러스터드 인덱스

테이블 자체가 특정 키 기준으로 정렬된 형태로 저장되는 구조이다.

- PK 범위 조회에 유리하다
- PK 변경 비용이 크다

## 논클러스터드 인덱스

별도의 B+Tree 인덱스 구조를 가지며, 리프 노드에는 실제 레코드 위치를 가리키는 포인터가 저장된다.

- 다양한 조건 검색에 유리하다
- 인덱스에서 테이블로 다시 접근하는 추가 I/O가 발생할 수 있다

---

# 5. 인덱스가 있어도 느릴 수 있는 이유

## 선택도가 낮은 경우

조건에 해당하는 데이터 비율이 높으면 인덱스를 사용해도 많은 데이터를 읽게 되어 비효율적이다.

## 함수 사용

```sql
WHERE DATE(created_at) = '2026-03-03'
```

컬럼에 함수가 적용되면 인덱스를 제대로 활용하지 못할 수 있다.

## 복합 인덱스

복합 인덱스(a, b, c)는 왼쪽 컬럼부터 사용하는 것이 중요하다.

---

# 6. 인덱스 심화 개념

## 6-1. 커버링 인덱스 (Covering Index)

쿼리에서 필요한 모든 컬럼이 인덱스에 포함되어 있어 테이블을 다시 조회하지 않아도 되는 경우를 의미한다.

```sql
SELECT id, name
FROM users
WHERE id = 10;
```

id, name이 인덱스에 포함되어 있다면 추가 테이블 접근이 발생하지 않는다.  
랜덤 I/O를 줄이는 데 매우 효과적이다.

---

## 6-2. 선택도(Selectivity)

선택도는 전체 데이터 중 조건에 해당하는 데이터의 비율이다.

- 선택도가 높다 → 인덱스 효율 좋다
- 선택도가 낮다 → 인덱스 효율 낮다

예:

- 주민등록번호 → 선택도 매우 높다
- 성별 → 선택도 낮다

---

## 6-3. 인덱스의 단점

- INSERT 시 인덱스도 함께 갱신된다
- UPDATE 시 재정렬 비용이 발생할 수 있다
- DELETE 시 인덱스 정리 비용이 발생한다
- 저장 공간을 추가로 사용한다

읽기 성능 향상과 쓰기 성능 저하 사이에는 트레이드오프가 존재한다.

---

## 6-4. 실행계획(EXPLAIN)

DB는 비용 기반 옵티마이저를 통해 실행 계획을 결정한다.

```sql
EXPLAIN SELECT * FROM users WHERE id = 10;
```

확인해야 할 요소:

- type (ALL, index, range, ref 등)
- key (사용된 인덱스)
- rows (예상 스캔 건수)

인덱스 설계 후 반드시 실행 계획을 확인해야 한다.

# 7. 조인 알고리즘

## Nested Loop Join

한 테이블을 기준으로 다른 테이블을 반복 탐색하는 방식이다. 내부 테이블에 인덱스가 있으면 효율적이다.

## Hash Join

작은 테이블을 해시 테이블로 구성한 뒤 큰 테이블을 스캔하면서 매칭하는 방식이다. 등가 조인에 강하다.

## Sort Merge Join

양쪽 테이블을 정렬한 후 병합하는 방식이다. 정렬 비용이 발생한다.

# 8. 요약

- B-Tree와 B+Tree는 디스크 환경에서 탐색을 빠르게 하기 위한 균형 트리 구조이다
- B+Tree는 범위 검색에 매우 강하다
- 인덱스는 선택도와 조건 형태에 따라 성능이 달라진다
- 인덱스는 읽기 성능을 향상시키지만 쓰기 비용이 증가한다
- DB는 상황에 따라 다양한 조인 알고리즘을 선택한다
